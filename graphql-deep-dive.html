01
Endpoint Design
Single GraphQL endpoint replaces versioned REST routes
Critical Breaking change prevention
−
🔴 The Problem

The current architecture uses POST /api/v1/timeaway/{blah} as the endpoint contract. This creates rigid coupling between client and server.

  • Every new field or feature requires a new endpoint or a version bump (/v2/, /v3/)
  • Mobile and web clients receive the same response shape even if they need different data
  • Over-fetching: mobile gets 40 fields but only uses 8
  • Under-fetching: dashboard needs data from 3 endpoints, making 3 separate calls
  • API contracts are implicit — no compile-time guarantees for consumers
🟢 The Solution

GraphQL exposes a single POST /graphql endpoint. Clients send a query document specifying exactly the fields they need — the server returns exactly that shape, nothing more.

  • Mobile query requests 8 fields; web query requests 40 — same endpoint, different shapes
  • New fields are additive — zero breaking changes, zero versioning
  • Deprecated fields marked in SDL, removed only when all consumers stop using them
  • Dashboard fetches absence + balance + history in a single query
  • SDL type system gives compile-time guarantees via codegen

02
Schema Validation
SDL type system eliminates the schema cache round-trip
Critical −4 steps removed
+
🔴 The Problem

The original diagram shows 4 of 11 steps are devoted entirely to schema fetch and validation: steps 1, 2, 10, 11. The Schema Cache is an external runtime dependency that adds latency on every request path.

  • Step 1: BFF fetches schema for outgoing request from Schema Cache
  • Step 2: BFF validates request against retrieved schema
  • Step 10: BFF fetches schema for response from Schema Cache
  • Step 11: BFF validates response against retrieved schema
  • Schema Cache becomes a SPoF — if it's down, requests fail
  • Cache invalidation lag means outdated schemas can silently pass invalid payloads
🟢 The Solution

GraphQL's SDL is intrinsic to the server — it IS the server. Type checking happens at query parse time, before any resolver fires. There is no external schema dependency to fetch or validate against.

  • SDL is parsed once at server startup — zero per-request schema fetches
  • Type validation happens during query parsing, not in a separate call
  • graphql-codegen generates TypeScript types at build time — errors caught before deploy
  • Introspection endpoint replaces the Schema Cache for discoverability
  • Apollo Studio provides a live schema registry with change diffing
  • No SPoF: schema lives in the process, not a network dependency

03
Service Architecture
Apollo Federation — unified supergraph from domain subgraphs
Critical Team autonomy
+
🔴 The Problem

The Time Away Service is a monolithic REST service. Absence Management, Balances, and Active History live behind one shared interface. Any team touching any domain must coordinate deploys.

  • Single deploy pipeline for 3 independent domains
  • Shared data model — one team's change can break another's API contract
  • The BFF has no visibility into sub-domain structure — it sees one opaque service
  • Scaling one sub-domain (e.g. History) requires scaling the whole service
  • A bug in Balances can take down Absence Management
🟢 The Solution

Apollo Federation v2 lets each domain own a subgraph with its own SDL, resolvers, and deployment. The Federation Gateway stitches them into one unified supergraph. The client sees one schema; teams deploy independently.

  • Each subgraph owns its types — @key(fields:"id") links entities across subgraphs
  • Gateway performs query planning — knows which subgraph resolves which field
  • Subgraph schema changes go through rover subgraph check before publish
  • Independent scaling: History pod autoscales separately from Balances
  • Failure isolation: Balances subgraph errors don't cascade to Absence
Service architecture — federation
⚠ Old: Monolithic REST service
✓ New: Apollo Federation subgraphs
absence-subgraph.graphql — @key federation directive
# Each subgraph declares its owned types with @key
extend schema @link(url: "https://specs.apollo.dev/federation/v2.0", import: ["@key"])

type Employee @key(fields: "id") {
  id: ID!
  absences: [Absence!]!          # owned by this subgraph
}

# Balances subgraph extends Employee with its fields
type Employee @key(fields: "id") {
  id: ID!
  currentBalance: LeaveBalance   # owned by balances subgraph
}
04
Data Fetching
DataLoader — eliminates the N+1 query problem
Critical Performance
+
🔴 The Problem

GraphQL resolvers are called per-node in the response tree. Without batching, fetching 20 employees' balances fires 20 separate database queries — one per resolver call. This is the N+1 problem.

  • Query asks for 20 employees with their balances → 1 query for employees + 20 for balances = 21 DB hits
  • Each resolver is unaware of sibling resolvers running in parallel
  • Database connection pool exhaustion under moderate load
  • Latency multiplies with list size — O(n) database calls
🟢 The Solution

DataLoader batches all balance.load(employeeId) calls that occur within the same event loop tick into a single SELECT * WHERE id = ANY($1) query. Identical IDs are deduplicated automatically.

  • 20 employees' balances → 1 batched DB query regardless of list size
  • Per-request cache: same ID requested twice returns the cached result
  • DataLoader lives in request context — no cross-request data leaking
  • Works across federated subgraphs — each subgraph has its own loaders
Database query pattern
⚠ Old: N+1 — 21 queries for 20 employees
✓ New: DataLoader — 1 batched query
dataloader.ts — batch function
import DataLoader from 'dataloader'

// Called ONCE per event-loop tick with all queued IDs
export const createBalanceLoader = () => new DataLoader<string, LeaveBalance>(
  async (employeeIds: readonly string[]) => {
    const rows = await db.query(
      'SELECT * FROM balances WHERE employee_id = ANY($1)',
      [employeeIds]   // all 20 IDs in one call
    )
    // Must return in SAME ORDER as input IDs
    return employeeIds.map(id =>
      rows.find(r => r.employee_id === id) ??
      new Error(`Balance not found: ${id}`)
    )
  },
  { cache: true }  // same ID in same request = cached
)

// In resolver — called for EVERY employee node
const resolvers = {
  Employee: {
    currentBalance: (emp, _, { loaders }) =>
      loaders.balance.load(emp.id)  // batched automatically
  }
}
05
Real-time
GraphQL Subscriptions replace polling for live balance updates
High impact Mobile battery & bandwidth
+
🔴 The Problem

When a manager approves time off, the employee's leave balance changes. The mobile app has no way to know this without repeatedly polling the balance endpoint — wasting battery, bandwidth, and server resources.

  • Mobile polls GET /balance every 30 seconds hoping for changes
  • 99% of polls return the same data — wasted requests
  • Battery drain on mobile from wake-up cycles
  • Server load scales with number of active mobile users
  • Balance shown to user can be 30 seconds stale
🟢 The Solution

The approveAbsence mutation publishes to a PubSub channel. The mobile app holds an open WebSocket subscription. The server pushes the updated balance the instant approval completes — zero polling.

  • WebSocket/SSE connection idle until a real event occurs
  • Mutation publishes to BALANCE_UPDATED:{employeeId} channel
  • Only subscribed employee receives the push
  • Falls back to SSE (Server-Sent Events) for environments without WebSocket
  • Balance shown to user updates in under 100ms after approval

⚠ Old: Polling every 30s
✓ New: Subscription push on approval
06
Mobile Performance
Automatic Persisted Queries — reduce mobile payload on repeat calls
High Mobile bandwidth
+
🔴 The Problem

Every GraphQL request sends the full query string in the POST body. A typical Time Away query document can be 400–800 bytes. On mobile with poor connectivity, this adds latency and costs data.

  • Full query string sent on every request even if it never changes
  • Cannot be GET-cached by CDN (body-based queries aren't cacheable)
  • Large queries with fragments amplify the waste
🟢 The Solution

APQ (Automatic Persisted Queries) sends only a SHA-256 hash of the query on subsequent calls. On first hit the server returns a PersistedQueryNotFound error; the client re-sends with the full query. All subsequent calls send only the 64-char hash.

  • Repeat queries: 64-byte hash vs 600-byte query string — 90%+ reduction
  • GET-request compatible — CDN can cache by hash as URL parameter
  • Apollo Router supports APQ out of the box
  • Works transparently — no changes required in resolvers

✓ New: APQ — hash after first call
07
Security
Query depth and complexity limits — protect against expensive queries
High Security
+
🔴 The Problem

GraphQL's flexibility is its attack surface. A malicious or accidentally recursive query can nest deeply and trigger exponential resolver calls, exhausting your database connection pool and bringing down the service.

  • Deeply nested query: employee{absences{employee{absences{employee{...}}}}}
  • Each nesting level can multiply resolver calls by the list size
  • No REST equivalent — GET requests have fixed cost; GraphQL does not
  • Without limits, a single request can simulate a DDoS
🟢 The Solution

Enforce two limits at the Apollo Gateway/Router level before any resolver fires: a maximum query depth (8 levels) and a complexity budget (500 points) where each field has a cost and lists multiply their contents.

  • Max depth 8 covers all legitimate Time Away queries (actual max ~4)
  • Complexity budget: scalar = 1pt, list = 10pt × child cost
  • Limits enforced in the query parser — resolvers never fire for invalid queries
  • Apollo Router plugin: @cost(weight: 10) directive on list fields
  • Rate limiting per employeeId claims from JWT
Query complexity enforcement
⚠ Old: Unbounded query depth
✓ New: Depth + complexity enforced at gateway
08
Schema Governance
Schema registry becomes SDL contract versioning — CI/CD gate
Medium DevOps
+
🔴 The Problem

The Schema Registry in the current diagram is a runtime dependency — queried on the hot path during every request (steps 4, 7). It's a network call inside the request lifecycle, adding 10–50ms per call.

  • Runtime schema fetch adds latency to every request (steps 4, 7)
  • Schema Registry downtime causes request failures
  • Schema changes aren't validated before deploy — only discovered at runtime
  • No diff history — hard to know when a field was added or removed
🟢 The Solution

The registry keeps its SDLs but moves entirely off the hot path. Apollo Studio runs rover subgraph check in the CI pipeline before any deploy. Breaking changes fail the pipeline — they never reach production.

  • rover subgraph check --schema absence.graphql runs in CI
  • Breaking changes (removed fields, type changes) fail the pipeline
  • Non-breaking additions (new fields) pass automatically
  • Schema changelog visible in Apollo Studio
  • Zero runtime dependency — registry is a deploy-time tool, not a request-time tool
Schema registry — runtime vs deploy-time
⚠ Old: Registry on hot request path
✓ New: Registry as CI gate only
09
Observability
Field-level tracing and resolver performance monitoring
Medium Observability
+
🔴 The Problem

REST monitoring gives you request-level metrics: status codes, latency, error rates. But when a GraphQL query is slow, you need to know which resolver field caused it — something REST logs cannot tell you.

  • Slow requests look the same in REST logs regardless of which resolver is at fault
  • No visibility into which fields are actually being used by clients
  • Cannot deprecate a field safely — you don't know if anyone still queries it
  • N+1 problems may exist silently — no per-resolver DB call count
🟢 The Solution

Apollo Studio's field-level tracing instruments every resolver automatically. You see a flame graph per query showing which resolver took how long, which fields are queried by which clients, and which fields have been unused for 30 days.

  • Resolver flame graph: see exactly which field is slow
  • Field usage analytics: know if a deprecated field is still called before removing it
  • Operation signatures: group all queries by shape for aggregate analytics
  • Error tracking: per-field error rates, not just per-endpoint
  • Apollo Router plugin emits OpenTelemetry spans per resolver — Jaeger/Datadog compatible
