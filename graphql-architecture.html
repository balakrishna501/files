Persisted queries — payload reduction

On mobile, full query strings sent over the wire every request are expensive. APQ (Automatic Persisted Queries) sends a SHA-256 hash on repeat calls. Cache miss? Server fetches and stores the full query once.

APQ SHA-256 hash CDN cacheable

Critical

DataLoader — solves N+1 in resolvers

Without DataLoader, fetching 20 employees' leave balances fires 20 separate DB queries per request. DataLoader batches all IDs within a single event loop tick and deduplicates identical lookups automatically.

batch + dedupe per-request cache event loop

Subscriptions — real-time balance push

Balance updates after approval currently require the mobile app to poll. A GraphQL subscription pushes the updated balance the instant the mutation completes — WebSocket or SSE, zero polling overhead.

Schema registry — SDL versioning only

The registry survives but its role shrinks: instead of validating every runtime payload, it stores SDL contracts for CI/CD schema-change detection. Tools like Apollo Rover catch breaking changes at deploy time, not at request time.

GraphQL's flexibility is also its attack surface. Unbounded nesting (absence → history → absence → ...) can exhaust resolvers. Enforce max depth (7–10) and a complexity budget per operation at the gateway level.

absence-subgraph.graphql — SDL type definitions
# Absence Management subgraph SDL
extend schema @link(
  url: "https://specs.apollo.dev/federation/v2.0"
  import: ["@key", "@shareable"]
)

type Employee @key(fields: "id") {
  id: ID!
  absences: [Absence!]!
  currentBalance: LeaveBalance
}

type Absence {
  id: ID!
  type: AbsenceType!
  startDate: Date!
  endDate: Date!
  status: ApprovalStatus!
  days: Float!
}

type LeaveBalance {
  annual: Float!
  sick: Float!
  remaining: Float!
}

enum ApprovalStatus {
  PENDING
  APPROVED
  REJECTED
}

type Mutation {
  submitTimeAway(input: TimeAwayInput!): Absence!
  approveAbsence(id: ID!): Absence!
}

type Subscription {
  # Real-time balance push on approval
  balanceUpdated(employeeId: ID!): LeaveBalance!
}
dataloader.ts — batching resolver example
import DataLoader from 'dataloader'
import { db } from './database'

// Batch all balance lookups within one event-loop tick
export const balanceLoader = new DataLoader<string, LeaveBalance>(
  async (employeeIds) => {
    // ONE query for N employees, not N queries
    const balances = await db.query(
      `SELECT * FROM leave_balances
       WHERE employee_id = ANY($1)`,
      [employeeIds]
    )
    // Return in same order as input IDs
    return employeeIds.map(id =>
      balances.find(b => b.employee_id === id)
      ?? new Error(`Balance not found: ${id}`)
    )
  },
  { cache: true } // per-request dedup
)

// Resolver — called for every Employee node
const resolvers = {
  Employee: {
    currentBalance: (employee, _, { loaders }) =>
      loaders.balance.load(employee.id),
      // Batched — fires once for all employees
  }
}

// Gateway: depth + complexity limits
const gateway = new ApolloGateway({
  maxDepth: 8,
  maxComplexity: 500,
  complexityEstimators: [fieldExtensionsEstimator()],
})
05 Design principles applied
01

Schema as contract, not documentation

SDL is the source of truth enforced at compile time by codegen tools (graphql-codegen). Type mismatches become CI failures, not runtime surprises. The schema registry becomes a diff tool, not a validation dependency.

Build-time safety
02

Subgraph per domain, not per team

Federation boundaries should follow bounded contexts — Absence, Balances, Active History each map cleanly. Avoid the antipattern of creating subgraphs for organisational convenience; they'll drift out of sync with domain models.

Domain integrity
03

Batch at the DataLoader, not the resolver

Resolvers should never call the DB directly. Every data access goes through a DataLoader that lives in the request context. This gives you automatic per-request deduplication and prevents thundering herd under load.

N+1 eliminated
04

Subscriptions are mutations' responsibility

When a mutation completes, it should publish the affected entities to a PubSub channel. Subscriptions subscribe to those channels. This decouples the push mechanism from the query/mutation resolver path entirely.

Real-time decoupled
05

Depth and complexity are non-negotiable

Every public GraphQL API — including internal BFFs that mobile apps hit — needs query depth and complexity limits. A depth of 8 and complexity budget of 500 covers all legitimate Time Away queries while blocking recursive amplification attacks.

Hardened API
